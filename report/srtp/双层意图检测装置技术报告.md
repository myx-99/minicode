## 1. 引言与背景

### 1.1 问题定义

在 AI Agent 系统中，用户的原始任务（goal）与 LLM 生成的具体执行步骤（plan step）之间可能出现以下对齐问题：

| 问题类型 | 描述 | 示例 |
|----------|------|------|
| **目标偏离** (Goal Deviation) | 步骤追求了错误的目标 | 用户要求"修复导入错误"，Agent 却"重新设计数据库 Schema" |
| **任务编排错误** (Task Orchestration) | 错误的分解方式或跳过必要子步骤 | 直接在未定位错误的情况下重写整个文件 |
| **资源滥用** (Resource Abuse) | 冗余/重复的工具调用 | 连续读取同一个文件 10 次 |
| **上下文忽略** (Context Handling) | 忽略先前的计划和上下文 | 忽略前一步已经成功完成的修改 |
| **信息检索不足** (Poor IR) | 跳过必要的检索步骤 | 不搜索相关代码就直接修改 |

本报告描述的**意图检测装置**旨在实时审计 Agent 的每一步行动，判断其是否直接服务于用户目标。

### 1.2 设计动机

传统的单层审计方案依赖 LLM-as-Judge 对每一对 (goal, plan_step) 进行 NLI（Natural Language Inference）判断。这种方法存在以下问题：

- **延迟高**：每次调用 LLM 需要 1,000–2,000ms
- **成本高**：每次调用消耗 LLM token（约 300 tokens/次）
- **稳定性差**：LLM 输出可能不稳定，相同输入可能得到不同判断

双层架构的核心思想是：**大多数 goal-step 对在语义空间中明显对齐或明显不对齐，只有少数边界情况需要 LLM 的深度推理能力**。通过嵌入模型进行快速预筛选，可以大幅减少 LLM 调用次数。

## 2. 系统架构设计

### 2.1 总体架构

```
用户目标 (Goal)
    │
    ▼
┌─────────────────────────────────────┐
│         Layer 1: 嵌入过滤器           │
│  (Embedding Filter)                 │
│                                     │
│  Goal ──→ Embed API ──→ 向量 g      │
│  Step ──→ Embed API ──→ 向量 s      │
│                                     │
│  cos_sim(g, s) = g·s / (|g|·|s|)   │
│                                     │
│  若 < 0.48 → contradiction (跳过LLM) │
│  若 > 0.72 → entailment   (跳过LLM) │
│  否则     → 进入 Layer 2            │
└───────────┬─────────────────────────┘
            │ 灰色地带 (ambiguous)
            ▼
┌─────────────────────────────────────┐
│         Layer 2: NLI 法官            │
│  (LLM-as-Judge)                    │
│                                     │
│  使用 TRAIL 规划错误直觉框架          │
│  输出: entailment / neutral /       │
│        contradiction                 │
│  得分: 0.0 – 1.0                    │
└───────────┬─────────────────────────┘
            │
            ▼
      最终判定 + 分数
```

### 2.2 决策路径

```
空 Goal/Step → neutral (0.5)
embed_low=0.48, embed_high=0.72

cos_sim < 0.48       → contradiction (embed 路径)
cos_sim > 0.72       → entailment   (embed 路径)
0.48 ≤ cos_sim ≤ 0.72 → 进入 NLI Judge (nli 路径)
嵌入层异常            → 回退到纯 NLI (nli 路径)
```

### 2.3 嵌入模型选型

| 方案 | 模型 | 维度 | 延迟 | 费用 | 状态 |
|------|------|------|------|------|:--:|
| 云端方案 | DashScope text-embedding-v3 | 1024 | ~800-2500ms | $0.00002/次 | ✅ 当前使用 |
| 本地方案 | sentence-transformers/all-MiniLM-L6-v2 | 384 | ~5ms (预期) | 免费 | ⚠️ 待解决环境兼容性 |

## 3. 核心模块详解

### 3.1 模块结构

```
intent_auditor/
├── __init__.py              # 公共 API 导出
├── intent_auditor.py        # 单层 NLI 审计 (Layer 2)
├── embedding.py             # 嵌入提供者 (Layer 1)
└── two_layer.py             # 双层审计编排
```

### 3.2 NLI 法官 (intent_auditor.py)

**核心类**: `audit_intent(goal, plan_step) → IntentAuditResult`

采用 LLM-as-Judge 模式，System Prompt 包含 TRAIL 规划错误框架的五个维度，要求 LLM 输出结构化 JSON：

```json
{"label": "entailment|neutral|contradiction", "score": 0.0-1.0, "reason": "一句话解释"}
```

**决策逻辑** (`is_predicted_error`):
- `label == "contradiction"` → 预测为错误
- `score < threshold (默认 0.6)` → 预测为错误

**关键特性**:
- 支持 markdown 代码块中提取 JSON
- LLM 输出解析失败时回退到 neutral (保守策略：允许步骤继续)
- 分数自动 clamp 到 [0.0, 1.0]

### 3.3 嵌入过滤器 (embedding.py)

**核心抽象**: `BaseEmbeddingProvider` (抽象基类)

采用策略模式设计，支持自由切换后端：

```python
class BaseEmbeddingProvider(ABC):
    async def embed(self, text: str) -> EmbedResult      # 单条嵌入
    async def embed_batch(self, texts: List[str]) -> List[EmbedResult]  # 批量嵌入
```

**已实现的提供者**:

| 提供者 | 后端 | 特点 |
|--------|------|------|
| `DashScopeEmbeddingProvider` | 阿里云 text-embedding-v3 (OpenAI 兼容 API) | 1024维，中文友好 |
| `LocalEmbeddingProvider` | sentence-transformers 本地模型 | 384维，零网络开销 |

**余弦相似度计算**: 纯 Python 实现，无外部依赖：

```python
def cosine_similarity(a, b):
    dot = sum(x * y for x, y in zip(a, b))
    norm_a = math.sqrt(sum(x * x for x in a))
    norm_b = math.sqrt(sum(x * x for x in b))
    return dot / (norm_a * norm_b)
```

### 3.4 双层审计协调器 (two_layer.py)

**核心类**: `TwoLayerAuditor`

编排整个审计流水线：
1. 对 goal 和 plan_step 分别调用嵌入 API
2. 计算余弦相似度
3. 根据阈值决定路径
4. 嵌入层失败时自动回退到 NLI
5. 收集统计信息 (bypass rate, call counts)

**统计追踪**:
```python
self._stats = {
    "embed_hits": 0,   # 嵌入层直接判定的次数
    "nli_calls": 0,    # NLI 被调用的次数
    "total": 0         # 总审计次数
}
```

### 5.4 不一致案例分析

4 个标签不一致的 case:

| ID | 双层 → 单层 | 路径 | 分析 | 谁的判断更准确？ |
|----|-----------|------|------|:--:|
| high_01 | entailment → neutral | embed | "读 main.py 检查 imports" 明显是修导入错误的第一步。NLI 误判为 neutral | **双层** |
| high_09 | entailment → neutral | nli | "添加 argparse 参数" 直接对应"给 CLI 加参数"的目标 | **双层** |
| gray_05 | contradiction → entailment | embed | "查 git 提交记录排查 500 错误" — 是合理的调查步骤但被判为矛盾 | **单层** |
| gray_09 | contradiction → neutral | embed | "创建虚拟环境隔离测试以升级依赖" — 是安全实践，不应判为矛盾 | **单层** |

**结论**: 嵌入直判（embed 路径）导致的 2 个不一致中，1 个判断正确、2 个判断过于激进；NLI 路径导致的 2 个不一致中，2 个都是 NLI 判错。整体看，双层与单层的错误分布各有优劣，87% 一致性是可以接受的。

### 5.5 延迟分布分析

按路径分类的延迟分布：

| 路径 | 次数 | 平均延迟 | 最小 | 最大 | 组成 |
|------|:--:|------|------|------|------|
| embed (嵌入直判) | 16 | 1,442 ms | 988 ms | 3,216 ms | 2 次嵌入 API |
| nli (嵌入+LLM) | 14 | 3,140 ms | 2,729 ms | 3,756 ms | 2 次嵌入 API + 1 次 LLM |
| 纯 NLI (单层) | 30 | 1,944 ms | 1,612 ms | 4,791 ms | 仅 1 次 LLM |

**关键发现**:
- embed 路径：1,442ms（两次嵌入 API）
- nli 路径：3,140ms（两次嵌入 + 一次 LLM）— **嵌入成了纯额外开销**
- 纯 NLI：1,944ms（仅一次 LLM）— **反而最快**

根因：DashScope 嵌入 API 延迟 (~1,000-1,500ms/次) 与 DeepSeek NLI 调用 (~1,000-2,000ms) 在同一数量级。embed 路径需要 2 次嵌入调用，其总延迟并不显著低于 1 次 NLI 调用。

## 6. 瓶颈分析与改进方向

### 6.1 核心瓶颈：嵌入 API 网络延迟

```
理论预期:
  Embedding (本地):   ~5ms
  NLI (LLM):       ~1,800ms
  → embed bypass 节省 99.7% 延迟

当前实际:
  Embedding (DashScope): ~1,200ms/次
  NLI (DeepSeek):       ~1,800ms/次
  → embed bypass 仅节省 33% 延迟
  → 且需要 2 次嵌入，最坏情况反而更慢
```

### 6.2 改进路线图

| 优先级 | 改进项 | 预期效果 | 实现难度 |
|:--:|------|------|:--:|
| **P0** | 切换到本地嵌入模型 (sentence-transformers) | 延迟降低 99%+，费用降为 0 | 中（需解决 Python 3.13 + Windows segfault） |
| **P1** | 调优 DashScope 高阈值（0.72 → 0~0.68） | 高相似命中率从 40% 提升至约 60% | 低（修改配置） |
| **P1** | 实现本地 LRU 缓存（进程内 dict） | 同一 goal 重复出现时省去嵌入调用 | 低（~20 行代码） |
| **P2** | 支持批量嵌入优化（单次 API 同时发 goal+step） | 减少 TCP 连接建立开销 | 中 |
| **P2** | 自适应阈值（根据嵌入模型动态调整） | 自动适配不同模型 | 中 |

## 8. 结论

### 8.1 主要贡献

1. **设计并实现了双层意图检测架构**：嵌入过滤器（Layer 1）+ NLI 法官（Layer 2），提供策略模式的无缝后端切换。

2. **完成了系统性的 Benchmark 对比**：30 个覆盖三类语义区域的测试用例，同时对比双层和单层两种审计路径。

3. **证明了嵌入模型在意图检测中的有效性**：
   - 低相似区域命中率 **90%**，准确识别明显偏离的步骤
   - LLM 调用减少 **53%**
   - API 费用降低 **40%**
   - 标签一致率 **87%**（与纯 NLI 基本持平）

4. **识别了当前架构的性能瓶颈**：DashScope 云端嵌入 API 的网络延迟（~1,200ms/次）抵消了双层架构理论上应提供的延迟优势。

### 8.2 当前局限

- 嵌入 API 延迟与 LLM 调用延迟在同一量级，导致总延迟未改善
- 高相似区域的命中率仅 40%，阈值需要进一步调优
- 嵌入层的 2 次独立网络请求（goal + step）存在优化空间
- 本地嵌入模型因 Python 3.13 兼容性问题暂未启用

### 8.3 最终建议

**短期（可用）**：保持当前 DashScope 双层配置，关注 LLM 费用节省（53% 调用减少）和低相似区域的优秀过滤效果。

**中期（推荐）**：解决 Python 环境兼容性问题，切换到 `sentence-transformers/all-MiniLM-L6-v2` 本地模型。预期延迟降低 99%，费用降至 0，双层架构的全部优势将得以实现。

**长期（优化）**：实现自适应阈值、批量嵌入优化、进程内 LRU 缓存，进一步优化性能和资源使用。
